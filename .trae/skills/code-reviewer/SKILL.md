# 代码审查器

此技能用于对遵循项目规范和最佳实践的Go项目进行全面的代码审查。

## 调用时机

- 用户要求代码审查
- 完成重要代码更改后
- 合并代码更改前
- 用户想要检查代码质量

## 审查清单

### 1. 代码风格和格式

- [ ] 变量、函数和类型名称是否清晰并遵循Go规范？
- [ ] 是否有不必要的代码注释或缺少关键注释？
- [ ] 代码是否使用`gofmt`格式化？

### 2. 错误处理

- [ ] 错误是否正确检查（`if err != nil`）？
- [ ] 错误消息是否有足够的上下文（例如：`fmt.Errorf("xxx: %v", err)`）？
- [ ] `panic`和`recover`是否使用得当（避免滥用）？

### 3. 并发安全

- [ ] 对共享变量的访问是否使用互斥锁（`sync.Mutex`）或原子操作保护？
- [ ] 通道是否正确用于goroutine同步？
- [ ] 是否存在潜在的死锁、竞态条件或goroutine泄漏？
- [ ] `sync.WaitGroup`和`context.Context`是否正确用于并发控制？

### 4. 性能和资源管理

- [ ] 是否有不必要的内存分配（例如：使用`+`连接字符串而不是`strings.Builder`）？
- [ ] 资源是否正确关闭（文件、网络连接、HTTP响应体）？适当时使用`defer`。
- [ ] 循环内的重复计算是否可以提取到外部？
- [ ] 是否可以使用`sync.Pool`减少对象分配？

### 5. 代码组织和可读性

- [ ] 函数是否过长？是否可以拆分为更小的函数？
- [ ] 是否遵循"提前返回"原则以减少嵌套？
- [ ] 包结构是否清晰且职责单一？
- [ ] 全局变量和包级状态是否使用得当？

### 6. 测试

- [ ] 测试中的错误是否正确处理（使用`t.Fatal`而不是`panic`）？

### 7. 常见陷阱

- [ ] 是否在循环内使用`defer`？（可能导致资源释放延迟，考虑使用匿名函数）
- [ ] 是否错误地复制了不可复制的类型，如`sync.Mutex`、`sync.WaitGroup`？
- [ ] 是否错误使用`time.After`导致内存泄漏？

### 8. 业务逻辑检查

- [ ] 客户端请求消息字段是否完全验证？
  - 整数字段：负值、过大的值
  - 数组：空检查
- [ ] 数值溢出问题：int32算术运算是否会超过最大/最小值？
- [ ] 除零问题：分母是否可能为零？
- [ ] 购买/兑换逻辑：先扣除成本，再发放物品
- [ ] 数组索引越界风险
- [ ] 空指针解引用风险

## 审查流程

1. **阅读代码文件**以理解实现
2. **检查每个项目**在审查清单中
3. **识别问题**并按严重程度分类：
   - **严重**：安全漏洞、数据损坏、崩溃
   - **重要**：逻辑错误、资源泄漏、并发问题
   - **次要**：代码风格、性能优化、可读性
4. **提供反馈**，包括：
   - 问题描述
   - 文件位置和行号
   - 建议的修复或改进
   - 适用时提供代码示例

## 问题示例

### 空指针检查

```go
// 错误
hero := bagHero.GetElem(req.GetId())
hero.Level += 1  // 潜在的空指针解引用

// 正确
hero := bagHero.GetElem(req.GetId())
if hero == nil {
    l.Error("hero not found", "heroId", req.GetId())
    return nil, errors.New("heroNotFound")
}
hero.Level += 1
```

### 整数溢出

```go
// 错误
total := a + b  // 可能溢出int32

// 正确
if int64(a) + int64(b) > math.MaxInt32 {
    return errors.New("overflow")
}
total := a + b
```

### 资源管理

```go
// 错误
file, _ := os.Open(path)
data, _ := io.ReadAll(file)

// 正确
file, err := os.Open(path)
if err != nil {
    return err
}
defer file.Close()
data, err := io.ReadAll(file)
if err != nil {
    return err
}
```

### 业务逻辑顺序

```go
// 错误：先发放，后扣除（可能导致漏洞）
bag.AddItem(item)
bag.DelItems(cost)

// 正确：先扣除，后发放
if !bag.IsEnough(cost) {
    return errors.New("notEnough")
}
bag.DelItems(cost)
bag.AddItem(item)
```

## 注意事项

- 专注于可操作的反馈
- 优先考虑安全性和正确性而非风格
- 考虑项目现有的模式和规范
- 提供问题的上下文说明
- 建议具体的解决方案，而不仅仅是问题
